# Engineering Analysis

EA 是 PRD 和 spec 之间的开发草稿，用来记录开发者理解需求后的核心工程思路和设计草图。

EA 不负责兜全细节。详细接口、表结构、状态机、权限、测试、上线、监控和回滚由 spec 兜底。

## 流程位置

使用顺序：

```text
PRD -> EA -> Spec -> Plan
```

- PRD：澄清产品目标和需求问题。
- EA：固化工程理解、核心思想路径和核心设计草图。
- Spec：展开为可技术评审的完整方案，兼顾产品、测试、前端、后端、兼容性和上线风险。
- Plan：拆成具体代码改动和任务执行步骤。

## 输出位置

在当前项目下创建或更新 EA：

```text
docs/superpowers/ea/YYYYMMDD-<topic>.md
```

不要写死其他项目的绝对路径。

不要执行 `git add`、暂存、提交或创建 PR，除非用户明确要求。

## EA 应该包含

默认使用这些章节：

1. 我理解的问题
2. 当前系统认知
3. 核心思想
4. 核心设计
5. 关键取舍
6. 主要影响点
7. 预期结果
8. 留给 spec 展开的内容

如果某些章节不适合当前任务，可以合并或删减，但不要把 EA 扩写成 spec。

## 写作边界

EA 要写：

- 用工程语言重述需求
- 当前系统相关链路和限制
- 核心思想路径
- 核心设计草图
- 主要模块和职责
- 关键取舍
- 主要影响点
- 预期系统行为
- 明确哪些细节交给 spec

EA 不要写：

- 完整接口文档
- 完整表字段
- 完整状态机细节
- 全量一致性论证
- 完整测试矩阵
- 监控、上线、回滚细节
- 文件级代码改动计划

## 核心设计粒度

EA 必须包含核心设计，但只写到草图层。

好的 EA 核心设计示例：

```text
现有提交链路
  -> 提交成功后触发
  -> 核心服务判断
  -> 规则 / 状态 / 幂等
  -> 写入目标状态
  -> 记录最小排查信息
```

不好的 EA：

- 列出每个表字段
- 写完整接口 request / response
- 展开所有失败分支
- 写成压缩版 spec

## 执行流程

1. 阅读 PRD、需求澄清、已有 spec、代码或用户上下文。
2. 用自己的工程语言说明“这次到底要解决什么”。
3. 检查当前系统相关链路，避免凭空设计。
4. 写出核心思想和核心设计草图。
5. 标出关键取舍和主要影响点。
6. 把细节明确留给 spec。
7. 完成前自查：这还是开发草稿吗？如果像技术方案评审稿，就删细节。
